Phase 1 · Complete
CI/CD Pipeline + Observability Stack

The foundation. A working CI/CD pipeline that builds, tests, and deploys automatically — combined with a full monitoring stack that collects metrics, visualises them, and fires alerts when things go wrong.

⚙ CI/CD
Jenkins: 8-stage automated pipeline — checkout, unit tests, multi-stage Docker build, Git-SHA tagging, push to GHCR, deploy, health check, endpoint verify. Code doesn't go live without a passing health check.
🌐 Traffic
Nginx → FastAPI: Nginx is the only service exposed on port 80. FastAPI is isolated on the internal Docker network. Nginx waits for FastAPI to be healthy before accepting traffic.
📊 Metrics
Prometheus + Node Exporter: scrapes FastAPI /metrics every 15s for application metrics and Node Exporter for CPU, memory, disk. Grafana visualises everything.
🔔 Alerting
Alertmanager → Slack: alert rules fire based on uptime and error rate. Alertmanager handles deduplication, grouping, and routing.

Phase 2 · In Progress
Intelligent Alert Automation

Built on top of Phase 1. Alerts go through an intelligent pipeline — enriched with context, triggering automated remediation, verifying recovery, and escalating if they can't fix themselves. The system also monitors its own alert pipeline and self-reports when it fails.

🔎 Blackbox
Blackbox Exporter: probes HTTP endpoints externally every 15s. Detects if n8n, Grafana, or the app are reachable from outside. N8NDown alert fires if the automation layer goes down.
🤖 n8n
Alert enrichment + remediation: parses alert payload, adds Grafana URL and runbook link, maps instance to container name, restarts failed containers via Docker API, verifies recovery via Prometheus API, posts full timeline to Slack.
🔒 Proxy
Docker Socket Proxy: n8n calls Docker API through a proxy that exposes only container restart. Principle of least privilege applied to container access.
📡 Fallback
Dual delivery: critical alerts route to n8n AND slack-backup simultaneously. If n8n is down, Alertmanager delivers directly. Critical alerts are never silently dropped.

Phase 3 · In Progress
AWS Infrastructure with Terraform

Real cloud infrastructure in eu-central-1 Frankfurt, provisioned entirely through Terraform. Infrastructure defined as code — version controlled, reproducible, and auditable. Always plan before apply.

☁ Network
VPC 10.0.0.0/16 with public subnet 10.0.1.0/24 in eu-central-1a Frankfurt. Internet Gateway, route table directing all outbound traffic. eu-central-1 chosen for proximity and GDPR compliance.
🔒 Security
IAM user with scoped permissions. Security Group controlling ports 22, 80, 443. 4096-bit RSA key pair for SSH — private key never leaves the local machine.
💻 Compute
EC2 t3.micro Ubuntu 22.04 with public IP. Instance Metadata Service at 169.254.169.254 for instance self-discovery — instance-type, AZ, public IP queryable from inside.
📋 IaC
terraform plan before every apply — see exactly what changes before anything happens. State tracks reality. Next: remote state in S3 + DynamoDB locking.

Phase 4 · Planned
Kubernetes on EKS

Migrate the entire stack from Docker Compose to Kubernetes on Amazon EKS. Orchestration at scale — pods, deployments, services, liveness probes, and Alertmanager running in HA mode across multiple replicas.

⎈ Kubernetes
EKS — Amazon Elastic Kubernetes Service: migrate all containers to pods and deployments. Kubernetes liveness probes replace manual health checks. Pod crashes restart automatically.
📊 HA Monitor
Alertmanager HA cluster: three-node cluster with gossip protocol for state sharing. No single point of failure in the alert pipeline.
⚡ Autoscaling
Horizontal Pod Autoscaler: pods scale up under load, scale down when idle. No manual intervention required for traffic spikes.
🎓 Target
CKA Certification: Certified Kubernetes Administrator. Studying while building — not instead of building.
